Spec-Driven Feature Workflow

A repeatable process for turning product requirements into production code

Software engineering in 2026 is fundamentally different. AI has shifted the bottleneck from writing code to defining what to build. The teams that ship fastest are the ones with the clearest specs. This workflow turns product requirements into production-ready architecture and code — across iOS, Android, web, or any stack — through a structured 5-step collaboration between PMs, designers, engineers, and AI.

Start by creating a requirements spec.

Step 1: Define Flows

PM / Designer

The PM or designer defines the feature's user flows: which screens exist, what each screen does, and how they relate to each other (parent-child hierarchy). This is where product requirements live.

What to capture

  • Screen names — Give each screen a clear name (e.g., "ProductSelection", "Review", "Success")
  • Parent-child relationships — Which screen owns which (e.g., Hub → BusinessDetails, ProductSelection, Review)
  • Product requirements — Write from the user's perspective: "User can select a product from the available options"
  • Design references — Paste or drag Figma screenshots directly into each screen

Tips

  • Start with the happy path — edge cases come in Step 3
  • One requirement per line, keep them atomic
  • Name screens by their purpose, not their UI component
  • Screenshots are optional but incredibly helpful for the architecture proposal

Step 2: Add API Calls

Engineer

The engineer adds technical details that the PM may not know: which API endpoints each screen calls, when each call is triggered, and what happens with the response.

For each API call, specify

FieldExampleDescription
endpointGET /v1/profiles/{id}/productsHTTP method + path
triggeronAppearWhen the call fires (onAppear, buttonTapped, etc.)

Tips

  • Use the actual API paths from your backend — Claude Code can generate the networking layer from these
  • Include both read (GET) and write (POST/PUT/DELETE) endpoints
  • If a screen makes multiple API calls, add each one separately

Step 3: Edge Cases & Errors

PM + Engineer

This is where the spec goes from "demo-ready" to "production-ready". Together, the PM and engineer think through what can go wrong and how the app should behave in unusual situations.

Edge Cases

  • Empty states (no data, no results)
  • Offline/poor connectivity scenarios
  • Boundary conditions (max items, very long text, special characters)
  • Race conditions (user taps multiple times, response arrives after navigation)
  • Permissions denied (camera, location, notifications)

Error Handling

Error ScenarioHandling
Network timeoutShow retry button with error message
Invalid IBANInline validation error below the field
Session expiredRedirect to login
Server error (5xx)Full-screen error with retry

External Dependencies

Services, SDKs, permissions, feature flags, or shared state this screen depends on. These can be @Dependency properties or Factory-injected services.

  • Feature flags (e.g., FeatureFlag.acquiringEnabled)
  • Permissions (Camera, Location, Notifications)
  • Shared services (Analytics, Deep Link Router)
  • Third-party SDKs (Stripe, Onfido, etc.)

Step 4: Propose Architecture

Claude Code

With the spec complete, Claude Code analyzes the requirements and proposes a full architecture. For iOS, this means a @StateReducer view model tree. For Android, a ViewModel hierarchy. The proposal includes:

  • State properties for each screen
  • Actions — ViewActions (user interactions), OutputActions (signals to parent), LocalActions (internal)
  • Dependencies — services, use cases, and APIs each screen needs
  • Ownership types — always-present (let), deferred (created after API response), or lazy (created on user interaction)
  • Navigation — push destinations vs sheet presentations
  • Error handling — how each error maps to state and UI

The engineer reviews the proposed architecture in the tree preview editor, adjusting ownership types, adding missing actions, and fine-tuning dependencies.

Step 5: Generate Code

Claude Code

Once the architecture is approved, Claude Code generates production code from the spec — Swift, Kotlin, web, or any stack. Using the /feature-scaffold skill:

  • All view models / ViewModels with state, actions, and reducers
  • All views / Composables / UI components
  • Parent-child wiring and lifecycle management
  • Networking layer — service protocols, implementations, DTOs, error types (via /networkingLayer)
  • Configuration structs with localization keys
  • Navigation setup

The engineer reviews the generated code, makes adjustments, and ships.

JSON Spec Format

The requirements page exports a JSON spec that Claude Code consumes. Here's the structure:

{
  "featureName": "OnboardingToAcquiring",
  "outputPath": "Wise/OnboardingToAcquiringKit",
  "root": {
    "name": "OnboardingHub",
    "requirements": [
      "User sees a step-by-step onboarding flow",
      "Progress indicator shows current step"
    ],
    "apiCalls": [
      { "endpoint": "GET /v1/profiles/{id}/onboarding", "trigger": "onAppear" }
    ],
    "edgeCases": [
      "User closes app mid-flow and returns",
      "API returns partial data for some steps"
    ],
    "externalDependencies": [
      { "name": "FeatureFlag.acquiringEnabled", "type": "feature flag" }
    ],
    "errorHandling": [
      { "error": "Network timeout", "handling": "Retry button with cached state" }
    ],
    "children": [
      {
        "name": "BusinessDetails",
        "requirements": ["User enters business name and type"],
        "children": []
      },
      {
        "name": "ProductSelection",
        "requirements": ["User picks from available products"],
        "apiCalls": [
          { "endpoint": "GET /v1/products", "trigger": "onAppear" }
        ],
        "children": []
      }
    ]
  }
}

Getting Started

  1. Open the requirements page with your feature name.
  2. PM/Designer defines flows — add screens, requirements, and Figma screenshots.
  3. Engineer adds API calls, edge cases, error handling, and external dependencies.
  4. Click "Copy prompt for Claude Code" and paste into Claude Code in the transferwise-ios or transferwise-android repo with the /feature-scaffold skill.
  5. Review the proposed architecture in the tree preview editor. Adjust ownership, actions, and deps.
  6. Claude Code generates all production code — view models, views, networking, and navigation.
