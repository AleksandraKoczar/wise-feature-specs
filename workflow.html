Overview

AI is fundamentally changing how software gets built. In 2026, the bottleneck is no longer writing code — it's deciding what to build and how it should behave. The teams that move fastest are the ones that invest in clear specs upfront and let AI handle the translation to production code. This shifts the craft from typing to thinking: defining requirements precisely, catching edge cases early, and iterating on architecture in minutes instead of days.

Spec-driven development replaces the old cycle of ad-hoc Slack threads, blank-file starts, and slow handoffs between PM and engineering. Instead, we use a requirements HTML page as the single source of truth. PMs, designers, and engineers all contribute to the same document — each in their area of expertise. Once complete, it generates a prompt that Claude Code uses to scaffold the full architecture and code. The result: iteration cycles that used to take a sprint now take an afternoon.

Step 1: Define Flows and Requirements PM / Designer

Open the requirements HTML page in a browser. For each flow in the feature:

  1. Name the screen — e.g., "BusinessDetails", "ProductSelection", "BankAccountSetup"
  2. Set parent-child relationships — use the dropdown to nest child screens under a hub/root screen
  3. Write product requirements — describe what the screen does from the user's perspective
  4. Paste design reference — Cmd+V Figma screenshots directly into the screen card

Example: OnboardingToAcquiring

The root screen is OnboardingHub with 5 children:

ScreenRequirements
OnboardingHub rootWelcome screen, progress indicator, back navigation
BusinessDetailsEnter business name + registration number, select business type, inline validation, pre-fill existing data
ProductSelectionShow available products with pricing, toggle enable/disable, at least one required
BankAccountSetupEnter IBAN or select Wise balance, IBAN validation as user types
ReviewSummary of all details, each section editable, submit disabled until valid
SuccessConfirmation, CTA to dashboard, confetti animation

Step 4: Generate and Review Architecture Engineer

Once all sections are filled:

  1. Click "Copy prompt for Claude Code" in the requirements page
  2. Paste into a Claude Code session in the repo
  3. Claude Code will:
    • Analyze the spec JSON
    • Propose the full @StateReducer view model tree (state, actions, effects, ownership types)
    • Generate an interactive HTML tree preview
  4. Review the tree preview — check ownership types, parent-child relationships, action naming
  5. Adjust if needed (add/remove nodes, change ownership from "born ready" to "on demand", etc.)

What Claude Code considers from the spec

Spec FieldHow it informs the architecture
RequirementsState properties, ViewActions, screen content
API CallsDependencies, Effects, LocalActions for responses
Edge CasesGuard clauses, validation logic, empty state handling
External Dependencies@Dependency properties or Factory-injected services, feature flag checks
Error HandlingError state properties, retry effects, UI error presentation

Step 5: Generate Code Claude Code

After reviewing the architecture, Claude Code generates all files:

  • ViewModels — @StateReducer view models with State, Actions, reduce(), parent-child wiring
  • Views — SwiftUI views with NavigationStack, sheet/push navigation
  • Networking — Service protocols, repositories, use cases for each API call
  • Configuration — Localization key stubs

All files follow the feature-scaffold skill conventions (naming, IntegratedAction handling, subscribeToChild pattern, CancellableEffect enum).

JSON Format

The requirements page exports a JSON tree. Here's a condensed example for OnboardingToAcquiring:

{
  "featureName": "OnboardingToAcquiring",
  "outputPath": "Wise/OnboardingToAcquiring",
  "root": {
    "name": "OnboardingHub",
    "requirements": ["Welcome screen", "Progress indicator", "Back navigation"],
    "children": [
      {
        "name": "BusinessDetails",
        "requirements": ["Enter business name + registration number", "..."],
        "apiCalls": [
          { "endpoint": "GET /v1/profiles/{profileId}/business", "trigger": "onAppear" }
        ],
        "edgeCases": [
          "Business name contains special characters or emoji",
          "Registration number format varies by country"
        ],
        "externalDependencies": [
          { "name": "ProfileService", "type": "shared state" }
        ],
        "errorHandling": [
          { "error": "Network error loading business data", "handling": "Show retry button" },
          { "error": "Invalid registration number", "handling": "Inline validation error" }
        ],
        "children": []
      }
    ]
  }
}

Fields are optional — screens with no API calls or edge cases simply omit those keys.

Tips
  • Start with requirements, not API calls. The PM/designer should be able to fill in Step 1 without knowing any backend details.
  • Edge cases are cheap to write, expensive to miss. Spend 5 minutes per screen thinking about empty states, race conditions, and unusual inputs.
  • Error handling drives UX quality. "Show generic error" is not a strategy. Be specific: retry button? Inline message? Fallback content?
  • The JSON is portable. Export it, check it into the repo, share it in a PR description. It's the contract between the spec and the code.
  • Review the tree before generating code. It's much faster to rename an action or change an ownership type in the tree preview than to refactor generated code.

Getting Started

  1. Open the requirements page in your browser
  2. Fill in your feature's screens and requirements
  3. Click "Copy prompt for Claude Code" or use "Preview tree" for quick iteration
  4. In Claude Code, run the /feature-scaffold skill in transferwise-ios or transferwise-android to scaffold the architecture from your spec
  5. Review the generated tree, refine ownership types and actions, then let the skill generate all production code

The requirements page, SKILL.md, and companion Python scripts are in skill-catalog/skills/feature-scaffold/.
